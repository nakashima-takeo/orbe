# ship — auto:ship に乗る Orbe 特化版

> 前提: `auto:ship`・`auto:plan`・`auto:implement`・`auto:review` はメンテナのプラグイン環境にあるスキルで、このリポジトリには同梱しない。無い環境ではこのスキルは動かない。

このスキルは auto:ship をそのまま完走させず、各フェーズに **Orbe 固有の差分だけ**を重ねる。auto:ship を読み込むとその手順に強く引っ張られるので、**この枠組みを先に頭に入れてから**読み込む。

**最初の行動は一つ——`Skill` ツールで `auto:ship` を読み込むこと。**

読み込んだら、その手順を本スキルのフローとして引き取り、各フェーズに下記の差分を重ねる——フローと指揮者規律・共有定型（委譲規約・ベースブランチ検出・作業空間の隔離（worktree）・PR 作成・マージ手順・`auto:implement`／`auto:review` の委譲プロンプト雛形）は auto:ship に任せる。

spec の規約は **`docs/spec/README.md` に従う**（現状のみ・粒度）。最初に Read で読む。題材は GitHub Issue か直接指示。`$ARGUMENTS` が題材（無ければ Issue から選ぶ）。

## Orbe 固有の絶対ルール（auto:ship に重ねる stance）

- **詰めの合意まで実装しない。** 完了条件（what）と設計を `AskUserQuestion` で合意してから実装に進む。題材に無いことを発明・補完しない。
- **設計の重さは題材の重さに比例させる。** auto:plan への調査+設計の依頼を規模に見合わせる（軽微なら完了条件の確定だけ、重いものだけ対立 approach・破綻シナリオまで詰めさせる）。要件に無い拡張を盛らない（YAGNI）。
- **spec に理由を書かない。** `docs/spec/` は「今どうなっているか」だけ。**なぜそう実装したかの理由は PR 本文にだけ書く**。
- **機械検証の合格を「動いている」と称さない。** 検証は二段——機械検証（完了条件だけを渡した別 agent。「動いてほしい」で自分の実装を甘く見ないため）と、人間の実機確認。人間が動くアプリを確かめて承認するまで確定しない。
- **見た目は自動で design に合わせきる。** chrome の見た目を変えた作業は、実機確認（layer②・人間・挙動）の前に `visual-check`（layer①・自動・見た目）を通し、design 見本とのズレをゼロにする。
- **マージは確定の後。** 実機承認 → 確定（spec 最新化・PR 本文への理由記録）まで終えてからマージする。
- **spec コミットとコードコミットを混ぜない。** コードは `auto:implement` / `auto:review` が打つ。spec の更新は ship が打つ。PR には両方が乗る。
- **承認ゲートは3点。** auto:ship の2点（詰め合意・マージ承認）に **実機承認**を挿入する。この3点以外のフェーズ間では確認しない。

## auto:ship フローへ重ねる差分

各差分は auto:ship のどのフェーズに乗るかを冒頭で示す。

### 受付に — 題材を見極める

題材は2通り。**直接指示**（最も多い入口）はそのまま詰めへ進む——人間の指示が着手根拠。**GitHub Issue 起点**なら `gh issue view` で中身を読んでから詰めへ——過去の ship が完了条件・設計まで積んだ Issue なら、それを引き継ぎ足りない所だけ詰め直す。いずれも `docs/spec/README.md` を Read し、日付は `date '+%Y-%m-%d'`。

### 調査+設計に — 計画書を書かせ、合意ゲートで分岐

調査+設計は **auto:plan に乗る**（要件の不明点は auto:ship が取り次ぐ）。計画書は auto:plan の既定 `.claude.local/plans/<kebab>.md`（untracked・コミットしない）に、**完了条件（what）を明示**して `auto:implement` が単体で読めるよう自己完結に書かせる。

**詰め合意ゲート（＝auto:ship の計画承認ゲート・Orbe の1点目）で分岐する**。計画書を Read し `AskUserQuestion` で合意する。

- **積むだけなら**、合意済みの完了条件と設計を GitHub Issue に記録して終了（Issue 起点なら本文へ追記、直接指示なら `gh issue create`）。後続の ship が受付で拾う。
- **続けるなら**実装へ。

### 実装の引き渡しに — 計画書とビルド

implement 雛形に乗せつつ、Orbe 固有の差分だけ足す——**計画書は `.claude.local/plans/<kebab>.md`**、**ビルドは `docs/BUILD.md` に従わせる**。返ってきた変更要約だけを保持し、逸脱や検証失敗があれば内容を把握する。以降の検証・確定は PR ブランチ上で行う。

### レビューに続けて — 検証を挿入

auto:review に乗る。**加えて Orbe の検証を挿入する**。

- **機械検証**。完了条件（計画書の what）**だけ**を渡した別 `Agent` に、各条件を独立に確かめさせる。実装者の申告は渡さない。条件ごとに確かめ方を設計させ（テスト・スクリプト・出力バッファ・ログ照合…手段は問わない）、**合格 / 不合格 / 人間目視が必須**（理由付き）を返させる。不合格が1つでもあれば実装に戻す。
- **視覚検証（chrome の見た目を変えた作業のみ）**。`Agent`（subagent_type=general-purpose）を起動し、`visual-check` に chrome の見た目を design 見本とピクセル突合させ、ズレを直して push させる。layer①（自動・見た目）で、次の実機確認 layer②（人間・挙動）の前に通す。見た目を変えない作業ではスキップ。
  > プロンプト例:「`visual-check` スキルを使って、この PR の chrome（<対象>）の見た目を design 見本と突き合わせ、ズレがゼロになるまで直して push せよ。**対応要約（合わせた項目・残した逸脱と理由・コミット一覧）だけ**を返せ。」
- **実機確認（毎回必須・人間ゲート＝Orbe の2点目）**。`Skill` ツールで `sandbox-run`（承認モード）を呼び、新ビルドを本物の Orbe を止めない隔離した使い捨てインスタンスで起こして人間に承認させる。今回の変更が**どこに現れ・何を触って見るか**（人間目視が必須の条件も）を渡す。**この承認が確定・マージの許可**。NG・指摘があれば実装に戻す。

### マージの前に — 確定を挿入

**PR 本文に理由を記す**（完了条件・採用設計・捨てた approach と棄却理由・各完了条件をどう確かめ何が出たか＝検証の証拠・レビュー対応・実機承認）——spec の「なぜ今こうか」を後から辿る唯一の記録。実装で変わった現状を `docs/spec/` に反映する（粒度・記述レベルは規約に従う。該当 spec が無ければ新規作成）。spec の変更を1コミットにまとめ（`ship: <タイトル>`）、push する。これで PR が完成形になる。題材の Issue はマージで閉じる（`Closes #N` を PR 本文に）。

### マージの後に — ローカル反映

マージ後、`AskUserQuestion` で「自分の Mac の Orbe Dev を最新版に入れ替えるか」を問い、同意したら `Skill` ツールで `local-release` を呼ぶ（確定した main を常用の開発版 `/Applications/Orbe Dev.app` へ配達する最後の一押し。本番の `/Applications/Orbe.app` には触れない）。最後に、詰め・実装・検証・確定・マージ（・ローカル反映）の結果をまとめて報告する。
